
# 问题

大模型可直接将自然语言转换为SQL，非技术用户也能便捷查询数据库，在数据分析、商业智能等领域应用广泛。但目前存在一个亟待解决的关键问题：大模型生成的SQL往往语法正确、可正常执行，但语义与用户意图不符，返回错误结果而用户难以察觉，若应用于高风险决策场景将造成严重后果。这类"语法有效但语义偏离用户意图"的错误被称为**意图违反型幻觉**。

现有检测方法均存在明显局限性：
1.  基准评测方法需要预标注的ground-truth SQL作为对比基准，但实际生产场景中并不存在这类标注数据；
2.  静态/动态分析工具仅能检测语法错误、表/字段名不匹配等表层问题，无法判断生成的SQL是否符合用户语义意图；
3.  置信度估计类方法需要获取模型内部的token概率等信息，而商用大模型多为黑盒无法获取此类数据；且SQL生成任务中对同一问题多次采样的结果高度相似，SelfCheckGPT这类基于输出一致性的方法难以发现细微的语义偏差；
4.  LLM自评估方法在SQL检测场景中效果不佳，因为SQL正确性依赖精确的字段使用、关联逻辑和过滤条件，仅通过浅层语义匹配难以识别细微的意图偏差。

本文要解决的核心问题：**不用正确SQL做参考，也不用访问模型内部信息，怎么准确检测出大模型生成的SQL是不是真的符合用户意图？**


# 想法

针对NL2SQL的幻觉检测难题，作者提出两个核心思路：
1.  **通过问题改写验证结果一致性**：参考软件测试中"蜕变测试"的思路，无需知道正确答案，只需将用户问题进行语义受控的改写（例如将"列出2024年的订单"改写为"列出2023至2024年的订单"），若原始SQL正确，两次查询的结果应符合预期的逻辑关系（如后者应包含前者）。若结果不符合预期，则说明原始SQL存在错误。
    该方法比SelfCheckGPT这类反复询问同一问题的方案更有效，因为主动改写问题可引导模型采用不同的推理路径，更容易暴露隐藏的语义错误。实验表明该方式可使模型神经元激活覆盖率提升160%，确实能拓展推理路径的多样性。

2.  **针对常见错误模式设计检测规则**：LLM生成SQL的幻觉并非随机产生，存在固定的错误模式（如日期条件错误、表关联字段匹配错误等）。作者通过分析数千个错误案例，总结出常见错误类型，针对每类错误设计专门的问题改写规则，提升检测的针对性。


# 方案

MRSQLGen检测流程分为两个核心模块：**问题改写模块** 和 **结果交叉验证模块**，输入为用户问题与LLM生成的SQL，输出为"是否存在幻觉"的判断结果。

## 模块1：问题改写模块

1.  **构建幻觉知识库与分类体系（核心学术贡献）**：作者手动分析了3968个错误SQL案例，首次构建了系统性的NL2SQL幻觉分类体系，这是目前该领域最全面的错误模式梳理。
    整体分为四大类错误：
    - 语法错误（3.08%）：语法层面的错误，可被SQL解析器直接捕获
    - 语义错误（15.73%）：语法正确但执行时报错，如表不存在、字段不匹配等
    - **意图违反型幻觉（77.62%）**：语法和执行都正常，但语义不符合用户意图，是检测的核心难点，细分为10个子类：
      - C1：运算符误用（4.9%）
      - C2：Limit错误（6.5%）
      - C3：关联逻辑错误（38.7%，最常见错误）
      - C4：值规范错误（12.9%）
      - C5：字段选择错误（15.7%）
      - C6：条件逻辑错误（18.6%）
      - C7：聚合函数误用（4.5%）
      - C8：Distinct错误（14.7%）
      - C9：OrderBy误用（9.9%）
      - C10：GroupBy误用（15.9%）
    - 其他错误（3.58%）：格式错误等低频率问题
    基于上述分类构建结构化知识库，存储"问题特征-错误类型"的对应关系，为后续针对性检测提供基础。
2.  **预测当前查询的潜在错误类型**：
    - 对用户问题与生成的SQL进行标准化处理，去除无关词汇差异，保留核心语义结构；
    - 采用模型将问题与SQL编码为向量，从知识库中检索最相似的历史案例，通过相似度加权投票，预测当前查询最可能出现的错误类型。
3.  **针对性改写用户问题**：针对预测的错误类型，设计了4大类共26条针对性改写规则：
    - 意图扰动与反转：例如将"大于100"改为"小于100"、将"2024年"改为"2023年"，验证SQL能否正确响应意图变化，针对运算符误用、值规范错误等类型；
    - 逻辑分解：例如将"查找收入大于5000且居住在北京或上海的用户"拆解为"先筛选收入大于5000的用户，再过滤居住在北京或上海的用户"，引导模型逐步推理，针对条件逻辑错误、聚合函数误用等类型；
    - 约束显式化：例如将"第二页结果"明确为"每页10条，返回第11至20条记录"，避免模型理解偏差，针对Limit错误、OrderBy误用等类型；
    - 错误类型反射：例如提示模型"请注意检查日期条件是否正确"，引导模型规避常见错误，适用于所有幻觉类型。

## 模块2：结果交叉验证模块
- 将每个改写后的问题输入同一LLM生成SQL，并在相同数据库上执行；
- 验证每个改写后SQL的执行结果与原始结果是否符合预期逻辑关系（例如修改条件后结果应对应减少/增加/完全不同等）；
- 采用多数投票策略：仅当超过80%的改写查询结果不符合预期时，才判定原始SQL存在幻觉，降低大模型生成随机性带来的误判。



# 实验

论文围绕两个核心研究问题（RQ）设计实验，形成完整的证据链：


## RQ1：MRSQLGen检测意图违反型幻觉的效果是否优于现有方法？
**研究目标**：验证MRSQLGen相比现有检测方法是否有性能优势，是否具备通用性。
**实验设计**：
- 测试数据集：两个公开NL2SQL基准数据集
  - Spider：通用跨领域数据集，覆盖200个数据库，10181个查询
  - Bird：更复杂的真实业务场景数据集，包含更多复杂查询和业务逻辑
- 测试模型：5个主流大模型（GPT-4o、GPT-4o-mini、CodeLlama-13B、Deepseek-Coder-16B、Qwen2.5-Coder-14B），覆盖不同规模和厂商的模型，验证方法的通用性
- 对比基线：三类主流检测方法
  - SelfCheckGPT：基于多次采样结果一致性的检测方法
  - LLM自评估：让生成SQL的模型自己评估输出正确性
  - GPT-5.1外部裁判：用更强的大模型作为第三方评估正确性
- 评估指标：精确率（衡量误判率）、召回率（衡量漏判率）、F1值（综合效果）

**实验结果**：
MRSQLGen在所有模型和数据集上均显著优于现有基线方法：
- 在Spider数据集上，综合效果F1值较SelfCheckGPT提升112%~576%，较LLM自评估提升55%~430%，较GPT-5.1外部裁判评估提升21%~144%；
- 在更复杂的Bird数据集上提升幅度更大，例如采用CodeLlama时F1值达到**0.944**，较SelfCheckGPT提升181%，较GPT-5.1外部裁判评估提升94%；
- 统计显著性检验（p<0.05）表明所有提升均具有统计显著性，并非随机波动导致。


## RQ2：MRSQLGen的各个模块是否都对检测效果有正向贡献？
**研究目标**：验证幻觉知识库、针对性改写、多数投票等核心设计的必要性，分析各部分的贡献度。
**实验设计**：通过消融实验逐一移除核心模块，对比性能变化：
- 消融1：移除基于幻觉知识库的针对性改写，替换为通用问题改写，验证错误模式知识库和针对性规则的价值
- 消融2：移除多数投票策略，只要出现一个不一致结果即判定为幻觉，验证多数投票策略对鲁棒性的提升作用

**实验结果**：
- 移除幻觉知识库的针对性改写后，F1值在Spider数据集下降18.2%，在Bird数据集下降22.7%，证明基于错误模式的针对性改写是提升检测效果的核心因素，手动构建的分类体系和知识库带来了显著性能增益；
- 移除多数投票策略后，精确率下降37.5%，证明多数投票可有效降低大模型生成随机性带来的误判，是提升检测鲁棒性的重要机制。


# 思考

- **与相关工作的对比**：本文首次将"通过问题改写验证结果一致性"的思路应用于SQL幻觉检测任务，属于完全黑盒的检测方法，无需依赖ground-truth SQL作为参考，也无需访问大模型内部信息，在实际生产场景中适用性更强。核心创新在于将软件测试中的蜕变测试思想扩展到prompt层面，通过主动改写问题暴露推理不一致性，而非被动依赖多次采样对比。
- **局限性**：
  1. 每次检测需要执行多个SQL查询，在高并发生产环境部署可能带来额外的数据库性能开销；
  2. 目前的改写规则为人工设计，可能无法覆盖所有罕见错误类型；
  3. 若查询包含自定义函数、存储过程或复杂业务逻辑，检测效果可能受到影响。
- **未来延伸方向**：
  1. 该范式不仅适用于SQL生成任务，也可扩展至其他代码生成任务（如Python、Java）的幻觉检测，只需针对不同任务设计对应的改写规则即可；
  2. 可结合静态分析技术，先进行静态检查，减少实际执行的查询数量，降低数据库性能开销；
  3. 可探索自动学习改写规则的方法，降低人工设计成本。
- **研究启发**：对于结构化输出任务的幻觉检测，仅依赖语义分析效果有限，基于执行行为的一致性检测效果更显著；提前总结常见错误模式并进行针对性检测，效果显著优于通用方法。
